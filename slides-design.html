执行循环：ReAct vs Plan-and-Execute

两种主流执行循环模式的核心差异与适用场景

ReAct (Reasoning + Acting)

模仿人类解决复杂问题的直觉——一边推理，一边行动
Thought → 内部思考，分析当前状态
Action → 调用工具或向用户提问
Observation → 获取结果与反馈
Loop → 回到 Thought
灵活性极高
Token 消耗高
适用动态环境、探索性任务

Plan-and-Execute

目标明确、步骤清晰，对可靠性和可审计性要求高
Plan → 大模型制定完整执行计划
Execute → 轻量模型逐步执行
Verify → 验证结果，必要时重规划
灵活性较低
Token 消耗低
适用结构化任务、高可靠性
混合策略
在 Plan-and-Execute 框架下为复杂子任务嵌入 ReAct 循环——"大处着眼规划，小处着手应变"

执行模型：链式 → 图式 → 事件驱动

不同执行模型决定了 Agent 如何管理状态、处理分支和保持可恢复性
CHAINED

链式执行

线性流程：输入 → 步骤1 → 步骤2 → ... → 输出
✗ 无法回退
✗ 无法并行
✗ 无法暂停
✗ 恢复成本高
代表：早期 LangChain AgentExecutor
GRAPH / DAG

图式执行

控制流定义为有向图，支持环、分支和持久化
✓ 环(Cycle)——可回退
✓ 分支与并行
✓ 持久化检查点
✓ 人机协作(HITL)
代表：LangGraph StateGraph
EVENT-DRIVEN

事件驱动

通过消息传递和异步通信实现多 Agent 协作
✓ 多 Agent 协作
✓ 异步通信
✗ 抽象多，系统复杂
✗ 调试和迁移成本高
代表：AutoGen, MS Agent Framework

Claude Code 核心架构

QueryEngine
流式响应 · 工具调用循环 · 扩展思考 · 重试与 Token 计数
工具系统
~29K 行 · 原子性 · 可组合性 · 自我描述性 (Zod Schema)
上下文管理
三层压缩：子智能体隔离 → 上下文压缩 → 任务持久化
多智能体协调
6 种架构模式 · Worktree 文件系统隔离 · 独立消息数组
权限系统
默认/计划/自动批准/绕过/不询问 · 细粒度规则 · Git Worktree 沙箱

三层上下文压缩

L1 子智能体隔离 — 独立消息数组，不污染主会话
L2 上下文压缩 — 保留关键信息，丢弃冗余细节
L3 任务持久化 — 磁盘文件，跨会话恢复

Worktree 隔离机制

Git Worktree 为每个子智能体提供独立工作区，安全修改代码副本。任务完成后通过 Git Merge 合并回主分支。实现真正的文件系统级沙箱隔离。

错误恢复体系

429/529 自动重试 401 OAuth 刷新 413 上下文恢复 MaxToken 升预算 持续失败回退模型 流式中断→非流式续
